The Limited Role of Time

Time plays a limited role in the basic model described in the previous page.

Time: Before and After

When we prove that an algorithm is correct we make no assumptions about the times at which steps of the algorithm are executed. The only concept of time that we use in reasoning about program correctness is that some steps are executed before others.

before is a relation between steps that agents take in an execution of a system. before is defined by the following two rules:

  1. A message is sent before it is received.
  2. Each agent executes a sequence of steps. Steps earlier in the sequence occur before steps later in the sequence.

Example

Consider the following example of a receive. 
def receive(message, sender):
  send('a', agnt)
  send('b', agnt)
The time between the execution of the first and second send statements is unknown. We know that the first send occurs before the second. Equivalently, the second send occurs after the first.

before and after in Steps of an Agent

The model does not specify when a receive command of an agent is executed. An agent may have nonempty input channels at some point and may not execute a receive until later.

The model does not specify the time taken to execute a statement or the time that elapses between execution of successive statements during an execution of a receive function. The model allows for the possibility that an agent may be stopped while it is executing a receive and restarted some time later.

Agent Clocks may Drift by Arbitrary Amounts

An agent may have a local clock. A clock at one agent cannot be accessed by another agent. When reasoning about the correctness of an algorithm we make no assumptions about the accuracy of clocks of agents. We allow for the possibility that an agent's clock may drift from real time and from the clocks of other agents by arbitrary amounts.

Example

Consider the following example of a receive which is identical to that of the previous example except that a sleep statement has been inserted between the two sends. 
def receive(message, sender):
  send('a', agnt)
  time.sleep(1.0)
  send('b', agnt)
The receive functions in the two examples are equivalent when we reason about the correctness of algorithms. The presence or absence of the sleep command makes no difference to our proofs of correctness of algorithms. This is because we prove correctness while allowing for the possibility that clocks drift apart by arbitrary amounts.

Are Clocks Useful?

Clocks play an important role in the performance of algorithms. The performance of an algorithm in which an agent sleeps for a second is likely to be different from one in which the agent sleeps for a million seconds, even if clocks are not synchronized.

Later, we discuss algorithms for clock synchronization. GPS provides accurate timing. Time is important in distributed systems; however, if we can show that an algorithm is correct without depending on timing then we shall.
