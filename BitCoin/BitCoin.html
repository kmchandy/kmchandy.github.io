Introduction to BitCoin

BitCoin is based on cryptography and distributed consensus. We discussed the aspects of cryptography required for BitCoin in the previous module. This module focuses on distributed consensus. is a superb (and longer) description of the distributed consensus algorithm used in Bitcoin.

We discussed distributed consensus in the modules on Paxos and Byzantine Generals. The specification of consensus is weaker in BitCoin. Also, the BitCoin algorithms to achieve consensus are very different from the Paxos and Byzantine Generals algorithms.

No trusted agents

The previous module described a crypto currency system that required a trusted agent. Bitcoin manages without a trusted agent by transferring the trusted agent's responsibilities to the community of all agents who collectively make decisions. Specifically, the community determines how the ledger is modified, how transactions are verified, and how coins are generated.

No assumptions about numbers of agents

The Byzantine Generals algorithm uses an upper bound on the number of faulty agents and a lower bound on the number of non-faulty ones. Paxos assumes that the total number of agents is known. Bitcoin makes no assumptions about numbers of faulty and non-faulty agents. There are unlikely situations where the Bitcoin algorithm may not work, but we will ignore them. An example of such a situation is one in which all agents but one collude to prevent that one agent from executing transactions.

Ledgers may not match

A ledger is a record of a sequence of transactions. An advantage of having a single agent append values to the ledger is that all agents have a consistent copy of the ledger. Two agents may have different copies because one agent has a more recent one. The two copies can differ in only one way: the older version is a prefix of the more recent one, i.e., the more recent version consists of the older version plus additional elements. The older version will eventually catch up to the newer one.

In Bitcoin there is no single agent which is given the sole responsibility of appending values to the ledger. When multiple agents append values to a ledger concurrently, the ledger of one agent may not be a prefix of the ledger of another. So we can't rely on one ledger "catching up eventually" to another by appending more recent values. Bitcoin has a clever, and random, way of enabling ledgers to catch up to each other and become consistent eventually. Next we'll look at this concept of eventual consistency.
